你是一名擅長小說結構拆解的資深編輯。請基於“當前章節大綱上下文”進行階段劃分。

【全局信息】
- 全書總章節數：$total_chapters
- 當前已處理上下文塊數：{{item.chunk_index}} / {{item.total_chunks}}
- 全局最大階段數（硬上限，不是目標）：$max_stage_count
- 當前上下文建議階段上限（硬上限，不是目標）：{{item.max_stage_count_for_chunk}}
- 單階段大綱字數上限：6000 字

【當前上下文範圍】
- 章節範圍：第 {{item.start_chapter}} 章 ~ 第 {{item.end_chapter}} 章
- 章節數量：{{item.chapter_count}}

【硬約束（必須滿足）】
1. 輸出階段的 `chapter_start/chapter_end` 必須落在當前上下文範圍內，且按章節順序遞增、互不重疊。
2. 必須完整覆蓋當前上下文範圍：從 `{{item.start_chapter}}` 到 `{{item.end_chapter}}` 不能有任何漏章。
3. 不得輸出上下文範圍外章節；不得只覆蓋前半段後提前結束。
4. 不得與緊鄰章節形成“機械重複階段”（同名、同結構、僅改變1~2章範圍）。
5. 禁止輸出“極短重複階段”：如果某階段與前一階段同名，且內容高度相似，則必須併入前一階段，不得單獨輸出。
6. 禁止輸出僅 1 章的階段，除非該章出現明顯新目標/新衝突/新結構轉折，並在大綱中給出明確依據。

【數量約束（自適應，不要機械切分）】
1. 在不違背劇情自然性的前提下，遵循以下最低階段數：
   - 若 `{{item.chapter_count}}` <= 3：可爲 1 階段。
   - 若 `4 <= {{item.chapter_count}} <= 8`：至少 2 階段。
   - 若 `{{item.chapter_count}} >= 9`：至少 3 階段。
2. 階段數不得超過 `{{item.max_stage_count_for_chunk}}`。
3. 不要爲了湊數硬切；但也不能粗暴合併成“單階段覆蓋全塊”。

【質量要求】
1. 每個階段都要有明確“階段目標 / 主要衝突 / 關鍵轉折 / 階段結果”。
2. 階段名要簡潔有辨識度（例如“穿越與萌芽”），避免泛化命名。
3. 階段大綱要體現主角與主要角色變化，細節要可用於後續寫作指導。
4. 階段大綱必須“可展開”：至少覆蓋以下要素——階段起因、階段目標、衝突推進、關鍵事件鏈、角色關係變化、階段結果與懸念。
5. 階段大綱儘量詳細；必須寫成“帶小標題的結構化文本”，並嚴格包含以下內容：
   - 階段起因
   - 階段目標
   - 衝突與阻力
   - 關鍵事件鏈（至少 3 條）
   - 角色關係/能力變化
   - 階段結果與下一階段鉤子
   - 階段劇情概述（用400~800字流暢地概述這個階段的劇情）
6. 內容密度要高：每個小標題下至少 2 句信息，不得空話。
7. 每個階段 `stage_outline` 必須 ≤ 6000 字。

【輸出前自檢（務必執行）】
- 第一階段 `chapter_start` 是否等於 `{{item.start_chapter}}`？
- 最後一階段 `chapter_end` 是否等於 `{{item.end_chapter}}`？
- 相鄰階段是否無縫銜接（前一階段 end + 1 = 後一階段 start）？
- 是否存在“同名+高度相似+僅1章”的重複階段？若有，必須合併後再輸出。
