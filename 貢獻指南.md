# 貢獻指南

首先，感謝你願意爲 NovelForge 做貢獻 ❤️

無論是修復一個小問題，還是推進一個大功能，我們都非常歡迎。爲了讓協作更順暢、評審更高效，請優先遵循下面的約定。

## 1. 小功能 / Bug 修復：可以直接提 PR

適用於：

- 小範圍功能增強
- Bug 修復
- 文案與交互優化
- 小型重構（不改變核心架構）

提交 PR 時請務必包含：

1. **改動說明**：一句話說明這次改了什麼、解決了什麼問題。
2. **文件級說明**：列出本次改動的主要文件，並說明每個文件的改動目的。
3. **驗證方式**：你如何驗證改動有效（如手動步驟、關鍵日誌、測試結果）。

建議格式（可直接複製到 PR 描述）：

```markdown
## 改動概述
- 

## 文件改動說明
- `path/to/fileA`：
- `path/to/fileB`：

## 驗證
- 
```

## 2. 大功能：先提 Issue 討論，再實施

如果你計劃貢獻的是較大功能（例如涉及架構調整、核心流程改造、跨前後端的大改動），請先提一個 Issue，與作者/維護者討論後再開發。

Issue 建議包含：

- 背景與問題
- 目標與邊界（做什麼 / 不做什麼）
- 方案草圖（關鍵設計點）
- 對現有行爲的影響

這樣可以避免返工，也能幫助你更快獲得有效反饋。

### Issue 標題建議格式（中文簡潔版）

爲便於和普通問題反饋區分，建議使用以下中文標題格式：

- **大功能方案討論（推薦）**：`【方案討論】【模塊】一句話目標`
- **認領規劃功能（推薦）**：`【功能認領】【模塊】準備實現的功能`
- **小問題修復（可選）**：`【問題修復】【模塊】問題摘要`

示例：

- `【方案討論】【工作流】增加並行分支錯誤恢復策略`
- `【功能認領】【Agent】實現工作流 Agent 的批量示例模板`
- `【問題修復】【編輯器】修復章節正文自動保存衝突`

說明：

- `【方案討論】` 用於先討論方案，再進入開發；
- `【功能認領】` 用於“我準備做這個”，減少重複開發；
- 功能完成且 PR 被合併後，請關閉對應 Issue。

## 3. 開發規範：請遵守項目規則

開發時請遵守項目工程規則：

- `rules/novelforge-engineering-rule.md`

核心要求可以概括爲：

- 設計清晰，避免耦合和硬編碼
- 代碼整潔，優先可維護性
- 變更有校驗閉環，不做“看起來能跑”的脆弱實現

## 4. 代碼風格與協作建議

- 儘量保持“小步提交、聚焦改動”，避免一個 PR 混入過多無關改動。
- 不要順手重寫無關模塊（除非已在 Issue 中達成一致）。
- 優先補充或更新必要文檔，確保後續維護者能快速理解你的改動。

## 5. 溝通方式

如果你不確定某個設計是否合適，歡迎先開 Issue 或在 PR 中明確寫出你的取捨與疑問。

再次感謝你的貢獻！
