# NovelForge Engineering Rule

## 0) 目標

所有開發行爲以“長期可維護”爲第一優先級：

1. 低耦合（模塊邊界清晰）
2. 高內聚（單一職責）
3. 可驗證（變更有校驗閉環）
4. 可演進（避免一次性寫死）

禁止爲了“先跑通”引入難以回收的硬編碼和隱式行爲。

---

## 1) 架構與解耦規則

### 1.1 事件驅動優先

- 跨域聯動（例如“保存卡片後觸發工作流”）必須優先採用事件發佈/訂閱。
- 禁止在業務入口裏直接串聯多個下游模塊實現“硬調用鏈”。

### 1.2 插件化註冊優先

- 可擴展能力（初始化器、工作流節點、事件處理器）必須通過裝飾器註冊。
- 新增能力必須完成“兩步”：
  1) 定義並加裝飾器
  2) 在對應 `__init__.py` 導入，確保註冊生效

### 1.3 配置集中管理

- 所有可變參數必須進入統一配置體系（環境變量 + 配置對象）。
- 禁止在業務代碼中硬編碼：URL、開關、閾值、模型名、超時、重試次數。

---

## 2) 服務層與接口規則

### 2.1 單一職責

- 一個 service 只負責一個領域。
- 複雜能力拆分爲“小服務 + 協調層”，禁止“巨型上帝類服務”。

### 2.2 依賴注入

- 通過參數注入 `Session`、配置、依賴對象。
- 禁止在函數內部偷偷創建全局依賴實例並長期持有。

### 2.3 API 契約唯一來源

- 類型定義以後端 schema 爲唯一真相。
- 前端必須使用 OpenAPI 生成類型；禁止手寫重複接口類型並長期並存。
- 前端不要重複構建後端模型或接口類型，優先通過 `npm run gen:types` 從後端生成並複用。
- 任何 API 端點必須聲明清晰響應模型，確保契約可生成。

---

## 3) 工作流系統規則

### 3.1 代碼修改閉環

- 任何工作流代碼改動必須走“可驗證閉環”：
  1) 生成新代碼/補丁
  2) parse
  3) validate
  4) 通過後才應用

- 禁止跳過校驗直接落庫。

### 3.2 可視化編輯安全性

- 可視化參數修改不得直接拼接字符串寫回。
- 必須處理好字面量編碼/轉義，避免把合法值寫成帶多餘引號的錯誤代碼。
- 任何前端寫回都必須經過服務端校驗結果確認。

### 3.3 運行能力一致性

- 工作流功能設計必須考慮：
  - 後臺運行可見性（全局狀態反饋）
  - 節點級進度上報
  - 暫停/恢復
  - 運行記錄保留策略（臨時 vs 持久化）

- 新功能不得破壞以上能力的一致語義。

---

## 4) Agent 系統規則

### 4.1 共享層與業務層分離

- 通用消息渲染、流式事件、輸入框交互必須複用共享層。
- 業務 Agent 只實現自己的工具與策略，不復制一套 UI/協議。

### 4.2 單軌展示，避免雙實現

- 同一語義（如消息時間線）禁止維護兩套並行顯示邏輯。
- 若確需新增模式，必須證明舊模式無法滿足需求，並給出遷移計劃。

### 4.3 運行期依賴必須明確

- 關鍵提示詞等運行依賴缺失時應明確報錯。
- 禁止加入“只在源碼目錄可用”的隱式兜底讀取邏輯。

---

## 5) 前端工程規則

### 5.1 組件規模控制

- 超大組件必須持續拆分（UI、狀態、事件處理、數據轉換分離）。
- 優先抽離：共享組件、composable、類型定義。

### 5.2 暗黑模式與主題

- 樣式必須優先使用主題變量。
- 禁止硬編碼淺色文本導致暗黑模式不可讀。

### 5.3 狀態與事件一致性

- 任何流式交互都必須有明確狀態機（idle/running/stopped/error）。
- “發送/中止”必須共用一個動作入口，避免雙按鈕狀態衝突。

---

## 6) 數據與健壯性規則

### 6.1 輸入容錯與字段兜底

- 事件負載和節點輸入必須做必需字段防禦性處理。
- 對關鍵字段允許“多層回退解析”，避免因 `None` 直接中斷整條鏈路。

### 6.2 事務與異常邊界

- 失敗必須可回滾，不得產生半成功髒狀態。
- 對可降級功能（例如非關鍵聯動）可捕獲異常，但必須記錄結構化日誌。

---

## 7) 編碼與變更規則

### 7.1 最小變更原則

- 只改與目標相關的文件和邏輯。
- 禁止無關重構、無關命名重寫、無關格式化噪音。

### 7.2 編碼與文件安全

- 文本文件統一使用 UTF-8。
- 禁止在不確認編碼的情況下整文件重寫，優先局部補丁。

### 7.3 禁止反模式

禁止以下行爲：

- 爲“快”而複製粘貼同構邏輯（前後端、模塊間）
- 在多個位置維護同一份業務規則
- 用前端臨時字符串規則替代後端正式校驗
- 把過程性調試開關長期暴露給用戶而無產品意義

---

## 8) 驗證與交付規則

### 8.1 必做校驗

- 後端改動：至少做目標接口/流程級驗證。
- 前端改動：至少做關鍵路徑交互驗證（加載、提交、錯誤分支）。
- 工作流相關改動：必須驗證 parse/validate/run 的完整鏈路。

### 8.2 變更說明

- 交付說明必須包含：改動範圍、行爲變化、已驗證內容、已知限制。
- 如果存在暫未解決的邊界問題，必須顯式標註，不得隱藏。

---

## 9) 決策優先級（衝突時）

當多個方案都能實現功能時，按以下優先級決策：

1. 破壞性最小
2. 複用性最高
3. 校驗鏈最完整
4. 用戶體驗最一致
5. 實現複雜度適中

若以上無法兼顧，先保證正確性與可維護性，再優化體驗。
